# Рекомендация по лицензированию репозитория

## Что лучше сделать

### 1) Для кода
Использовать **Apache License 2.0**.

Почему:
- GitHub хорошо распознаёт эту лицензию;
- разрешает использование, изменение и распространение;
- требует сохранять copyright и лицензионные уведомления;
- даёт более серьёзную юридическую основу, чем просто текст в README;
- удобна для open-source Python-проектов.

### 2) Для prompt pack, документации и не-кодовых материалов
Использовать **CC BY 4.0**.

Почему:
- для не-кодового контента attribution формулируется понятнее;
- удобно для prompt library, гайдов, таблиц, шаблонов;
- можно явно требовать указание автора при использовании материала.

## Важный нюанс

Если ты хочешь, чтобы люди **обязательно указывали тебя как автора именно при любом использовании кода**, то в open-source мире это обычно означает:

- сохранение файла LICENSE;
- сохранение copyright notice;
- сохранение NOTICE, если он есть.

Но большинство популярных open-source лицензий **не заставляют** выводить твоё имя прямо в интерфейсе бота, в названии форка или в каждом публичном упоминании продукта.

То есть:
- **обязательное сохранение авторства в лицензии и notice** — да;
- **обязательная подпись “бот создан автором X” в UI каждого форка** — обычно нет.

Если тебе нужен именно жёсткий формат бренд-атрибуции в продукте, это уже лучше решать не стандартной open-source лицензией, а отдельным коммерческим соглашением.

## Практичная схема для GitHub

### В корне репозитория

- `LICENSE` → Apache-2.0
- `NOTICE` → краткое уведомление об авторе
- `README.md` → описание, что код под Apache-2.0
- `LICENSE-docs.md` или раздел в README → prompt packs / docs под CC BY 4.0

### Пример NOTICE

```text
Telegram AI AutoPoster
Copyright (c) 2026 Daniil

This product includes software originally created by Daniil.
Repository source and attribution notices must be preserved according to the LICENSE file.
```

## Что публиковать публично

Публично лучше оставить:
- код;
- шаблон `.env.example`;
- безопасный demo `prompts.example.json`;
- документацию;
- инструкции по установке.

Отдельно или приватно лучше хранить:
- чувствительные prompt packs;
- NSFW-наборы;
- реальные production-конфиги;
- токены и ключи;
- готовые сгенерированные файлы.

## Что я рекомендую именно для этого проекта

### Лучший вариант
- публичный GitHub-репозиторий с кодом и безопасной документацией;
- Apache-2.0 для кода;
- CC BY 4.0 для prompt packs, если они вообще публикуются;
- NSFW prompt bank не держать в публичном репозитории.

### Почему
Потому что публичный репозиторий с явным adult prompt content создаёт лишние риски для модерации, а кодовая база без откровенного контента выглядит как технический automation/generation tool и публикуется заметно спокойнее.
